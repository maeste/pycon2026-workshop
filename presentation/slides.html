→

We went from blaming the environment
to blaming the model temperature.

"Just ask it again with a better prompt" is the new
"Works on my machine — just reinstall Windows."

We solved "works on my machine" with containers and CI. Now we have a new excuse: "it worked in my prompt." Same dysfunction, new abstraction layer. We blame model temperature instead of dependency versions. We tweak system prompts instead of writing docs. The truth is — we never gave the agent what it needed in the first place. That's the problem we're going to fix today.

Three failure modes → Three axes to measure

Every horror story I hear maps to one of three failure modes. One: the agent doesn't understand what you want — no project context, no conventions. Two: the agent can't navigate — it can't find where tests go or what your entry point is. Three: the agent can't validate — no tests, no CI, so it has no feedback loop. Three failure modes is exactly why we'll measure three axes. The math isn't arbitrary.
KEY INSIGHT

It's Not the Agent's Fault

Agents aren't stupid.
They're uninformed.

An agent landing in your codebase is like a senior developer on day one.

Would you expect them to be productive without:
✅ A README explaining what the project does
✅ Clear coding conventions
✅ Knowledge of where things live
✅ Tests to verify their changes

No? Then why expect it from an agent?

Reframe: think of an agent as a senior dev showing up on day one. You wouldn't expect them to be productive without a README, without knowing your conventions, without tests. You'd give them a tour, point them at the wiki, pair with them for the first PR. We do all that for humans and then dump agents into the same repo with zero onboarding. Agents aren't stupid — they're uninformed.

The Three Questions

Q1: Does the agent understand WHAT we want?

Q2: Can the agent find WHERE things are?

Q3: Can the agent tell if it did RIGHT?

…and a 4th question we'll meet shortly: can the agent run SAFELY? 🛡️

Axis 4: SECURE

I submitted this talk with three axes. Building v2, one question kept surfacing that none of them answered:
"Can the agent run safely?"

Weight: 12%
📦

Sandbox & Isolation

Committed devcontainer, documented execution policy (LINCE / OS-sandbox / hosted)

🔑

Secret & Supply-Chain Hygiene

.gitignore secrets, .env.example, committed lockfiles, Dependabot

💉

Injection & Permissions

Instructions only in trusted files; restrictive agent deny rules (CVE-2025-59536)

Security & Sandbox = 12% · D6 · the only axis with a single dimension — for now.

Here's the plot twist. The talk title says three axes — that's how I pitched it. But building v2, security kept demanding attention that INSTRUCT, NAVIGATE, and VALIDATE didn't cover. An agent that understands, navigates, and validates perfectly can still leak secrets, run untrusted instructions, or get prompt-injected. So v2 adds a fourth axis: SECURE — can the agent run safely? It scores committed sandbox/isolation config, a documented execution policy (LINCE is one option among devcontainer, OS-sandbox, hosted), secret and supply-chain hygiene, and injection/permission hardening. It only rewards evidence in the repo — no unverifiable self-reports. 12 percent today, and I'd bet it grows.

Agent-Powered

The agent itself assesses your code — not a static script

Scoring in v2. The agent scans seven evidence-based dimensions — and here's a key v2 idea: every sub-criterion is tagged portable or target. Portable signals — AGENTS.md, tests, CI, lockfiles — count for any agent. Target signals — a CLAUDE.md bridge, vendor permission rules — only count when you declare an agent with --agents. No target declared? Those are marked n/a and excluded, so a portable repo isn't penalized for vendor files it doesn't need. The seven dimensions roll up into our four axes and gate against the maturity ladder. And it's the agent doing the reading — not a static script. That's what sets this apart from Kodus or Factory.ai.

CONTROVERSIAL TAKE

The Testing Paradox

Tests are NOT for finding bugs.

Tests are for giving AGENTS a safety net.

Without Tests
  1. Agent writes code
  2. You manually verify everything
  3. Bottleneck. Fatigue. Mistakes.
With Tests
  1. Agent writes code
  2. Agent runs tests → Pass ✓
  3. You review PR confidently

Tests multiply your effectiveness with agents.

Controversial take. Tests aren't primarily for finding bugs — they're for giving agents a safety net. Without tests, every PR routes through you for manual review. That doesn't scale. With tests, the agent writes code, runs the test suite, you only see green PRs. That's the multiplier. The way to scale with agents isn't to write more code yourself. It's to write the tests once and let the agent self-validate forever.

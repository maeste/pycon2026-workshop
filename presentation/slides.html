THE PROBLEM

"It Worked on My Machine"
→
"It Worked in My Prompt"

We went from blaming the environment
to blaming the model temperature.

"Just ask it again with a better prompt" is the new
"Works on my machine — just reinstall Windows."

We solved "works on my machine" with containers and CI. Now we have a new excuse: "it worked in my prompt." Same dysfunction, new abstraction layer. We blame model temperature instead of dependency versions. We tweak system prompts instead of writing docs. The truth is — we never gave the agent what it needed in the first place. That's the problem we're going to fix today.

Three failure modes → Three axes to measure

Every horror story I hear maps to one of three failure modes. One: the agent doesn't understand what you want — no project context, no conventions. Two: the agent can't navigate — it can't find where tests go or what your entry point is. Three: the agent can't validate — no tests, no CI, so it has no feedback loop. Three failure modes is exactly why we'll measure three axes. The math isn't arbitrary.
KEY INSIGHT

It's Not the Agent's Fault

Agents aren't stupid.
They're uninformed.

An agent landing in your codebase is like a senior developer on day one.

Would you expect them to be productive without:
✅ A README explaining what the project does
✅ Clear coding conventions
✅ Knowledge of where things live
✅ Tests to verify their changes

No? Then why expect it from an agent?

Reframe: think of an agent as a senior dev showing up on day one. You wouldn't expect them to be productive without a README, without knowing your conventions, without tests. You'd give them a tour, point them at the wiki, pair with them for the first PR. We do all that for humans and then dump agents into the same repo with zero onboarding. Agents aren't stupid — they're uninformed.

The Three Questions

Q1: Does the agent understand WHAT we want?

Q2: Can the agent find WHERE things are?

Q3: Can the agent tell if it did RIGHT?

Here's the model: three questions, three axes. WHAT, WHERE, RIGHT. INSTRUCT is the WHAT — what does this project do, what conventions do we follow. NAVIGATE is the WHERE — where does code live, how is it organized. VALIDATE is the RIGHT — how does the agent know it didn't break anything. Click any node on the triangle to jump to that axis. We're going to spend the rest of this workshop on each one in turn.

You can't skip levels — they're sequential.

Five maturity levels. L1 Foundational at 40 — you have a pulse. L2 Guided at 55 — there's basic context for agents. L3 Structured at 70 — agents can mostly work without supervision. L4 Optimized at 85 — you're in the top decile. L5 Autonomous at 95 — you could honestly let agents merge PRs. Most repos in the wild are L1 or L2. Today's goal: get you to L2 or L3 by the end of the workshop.

Hybrid Mode

Add LLM semantic scoring: 60% static + 40% AI judgment

Three-step scoring. Step one — static checks per axis. Does the file exist? Does the section exist? Is the tool configured? These are deterministic. Step two — sum the points earned, divide by max possible, get a 0 to 100 percentage. Step three — weighted average across the three axes, gate against the maturity ladder. Later we layer hybrid mode on top: 60 percent static plus 40 percent LLM judgment for content quality. We'll see that in Phase 4.

Run the scan with --axis navigate. Look at the output and pay attention to WHY each check matters, not just whether it passed. CONTROVERSIAL TAKE

The Testing Paradox

Tests are NOT for finding bugs.

Tests are for giving AGENTS a safety net.

Without Tests
  1. Agent writes code
  2. You manually verify everything
  3. Bottleneck. Fatigue. Mistakes.
With Tests
  1. Agent writes code
  2. Agent runs tests → Pass ✓
  3. You review PR confidently

Tests multiply your effectiveness with agents.

Controversial take. Tests aren't primarily for finding bugs — they're for giving agents a safety net. Without tests, every PR routes through you for manual review. That doesn't scale. With tests, the agent writes code, runs the test suite, you only see green PRs. That's the multiplier. The way to scale with agents isn't to write more code yourself. It's to write the tests once and let the agent self-validate forever.

The CI File 🔄

Copy. Adapt. Commit. Done.

.github/workflows/ci.yml
# Copy this template to your repo name: CI on: [push, pull_request] jobs: quality: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: { python-version: "3.12" } - run: pip install ruff pytest - run: ruff check . - run: ruff format --check test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 - run: pip install -e ".[dev]" - run: pytest tests/ -v --tb=short
+6 pts if present Adapt to your stack
Open templates/ci-template.yml. Copy it to .github/workflows/ci.yml in your repo. Adjust three things: Python version if you're not on 3.12, the linter line if you don't use ruff, and the test command if you don't use pytest. That's it — commit and push. You don't need anything fancier. A minimal CI that actually runs is infinitely better than a sophisticated one that doesn't exist. Seven minutes typing, two minutes to re-scan.
